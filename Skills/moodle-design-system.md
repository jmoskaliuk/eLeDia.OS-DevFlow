# Moodle Design System (MDS) ‚Äî Consumer & Contributor Reference

The Moodle Design System is published as **[@moodlehq/design-system](https://github.com/moodlehq/design-system)** on npm. It is a **React + TypeScript** component library built with **Vite library mode**, layered on top of **Bootstrap 5.3**. Tokens are managed externally via **ZeroHeight** and generated with **Style Dictionary**.

This skill is the authoritative source for any eLeDia plugin that consumes MDS. Where other skills (`moodle-framework.md`, `eledia-moodle-ux.md`) mention MDS, **this file overrides on conflict** ‚Äî the older skills were written against pre-release guesses.

> Package versions (as of repo snapshot 2026-04):
> - `@moodlehq/design-system` **2.1.1**
> - `react` / `react-dom` peer: **^19.2.4**
> - `bootstrap` dep: **^5.3.8**
> - Node: **‚â• v22.13.0**

## 1. Package layout & exports

The npm package exposes these entrypoints (from `package.json`):

| Import                                          | Resolves to                             | Use for |
| ----------------------------------------------- | --------------------------------------- | ------- |
| `@moodlehq/design-system`                       | `dist/index.js` (all components)        | App-shell / bundled consumers |
| `@moodlehq/design-system/css`                   | `dist/index.css` (tokens + comp styles) | One-time app CSS import |
| `@moodlehq/design-system/components/<name>`     | `dist/components/<name>/index.js`       | Selective loading (Moodle plugins) |
| `@moodlehq/design-system/tokens/css`            | `tokens/css/index.css`                  | Tokens-only (no components) |
| `@moodlehq/design-system/tokens/scss`           | `tokens/scss/_index.scss`               | Modern `@use` / `@forward` SCSS |
| `@moodlehq/design-system/tokens/scss/legacy`    | `tokens/scss/_index.legacy.scss`        | Core-LMS `scssphp` bridge (tag `MDS_LEGACY_SCSSPHP_COMPAT`) |

**Build architecture ‚Äî why it matters for Moodle:**

The build uses Rollup's `preserveModules` mode: every source file is transpiled in isolation into its own stable, hash-free output file. This is intentional ‚Äî Moodle loads files directly by URL without a consumer-side build step, so output paths must stay predictable across releases.

Component subpath imports are **auto-discovered** at build time. Adding a folder under `components/` is enough; no `package.json` or `vite.config.ts` edits needed.

### Two import styles (plugin code)

```js
// Full package ‚Äî brings all components (use if you already import the full CSS bundle)
import { Button } from '@moodlehq/design-system';

// Subpath ‚Äî JS for one component only; pair with a one-time app CSS import
import { Button } from '@moodlehq/design-system/components/button';
import '@moodlehq/design-system/css';
```

## 2. Tokens ‚Äî the `--mds-*` system

**The token prefix is `--mds-*`, not `--ds-*`.** Every CSS custom property is generated by Style Dictionary from DTCG JSON in `tokens/dtcg/`.

### Token categories

```
tokens/css/primitives.css   ‚Üí --mds-color-{hue}-{100..900}, --mds-scale-0..1800, --mds-typography-*
tokens/css/colors.css       ‚Üí semantic aliases (--mds-bg-*, --mds-text-*, --mds-border-*)
tokens/css/spacing.css      ‚Üí --mds-spacing-{none,xxs,xs,sm,md,lg,xl,xxl}
tokens/css/typography.css   ‚Üí --mds-font-family-*, --mds-font-size-*, --mds-font-weight-*,
                              --mds-line-height-*, --mds-letter-spacing-*
tokens/css/borders.css      ‚Üí --mds-stroke-weight-*, --mds-border-radius-*
tokens/css/sizes.css        ‚Üí --mds-size-*
tokens/css/shadows.css      ‚Üí --mds-shadow-*
tokens/css/breakpoints.css  ‚Üí --mds-breakpoint-*
```

### Naming convention

`--mds-{category}-{subcategory}-{modifier}`

| Category   | Real tokens (not guesses) |
| ---------- | -------------------------------------------------------------------------------------- |
| Background | `--mds-bg-interactive-primary-default`, `--mds-bg-interactive-primary-hover`, `--mds-bg-interactive-primary-active`, `--mds-bg-interactive-primary-disabled` |
|            | `--mds-bg-interactive-secondary-*`, `--mds-bg-interactive-danger-*` |
|            | `--mds-bg-surface-default`, `--mds-bg-surface-strong`, `--mds-bg-surface-subtle` |
|            | `--mds-bg-feedback-{primary,success,danger,warning,info,secondary}-{default,subtle}` |
| Text       | `--mds-text-default`, `--mds-text-muted`, `--mds-text-subtle`, `--mds-text-inverse`, `--mds-text-emphasis`, `--mds-text-danger` |
|            | `--mds-text-feedback-{success,danger,warning,info,primary,secondary}` |
|            | `--mds-text-link-primary-default`, `--mds-text-link-primary-hover` |
| Border     | `--mds-border-default`, `--mds-border-subtle`, `--mds-border-translucent` |
|            | `--mds-border-interactive-{primary,secondary,danger}-{default,hover,active,disabled}` |
|            | `--mds-border-radius-{none,xs,sm,md,lg,xl,xxl,pill}` |
| Spacing    | `--mds-spacing-{none,xxs,xs,sm,md,lg,xl,xxl}` (mapped to `--mds-scale-*`) |
| Stroke     | `--mds-stroke-weight-{sm,md,lg,xl,xxl}` |
| Typography | `--mds-font-family-base` (Roboto), `--mds-font-family-code` |
|            | `--mds-font-size-paragraph-{default,lead,small}`, `--mds-font-size-headings-{1..6}`, `--mds-font-size-displays-{1..6}` |
|            | `--mds-font-weight-{light,regular,medium,semibold,bold}` |
|            | `--mds-line-height-paragraph-{default,lead,small}`, `...-headings-{1..6}`, `...-displays-{1..6}` |
| Focus      | `--mds-focus-default` |

State variants for interactive tokens always follow: `-default`, `-hover`, `-active`, `-disabled`.

### Spacing & scale resolution

`--mds-spacing-*` all resolve through `--mds-scale-*` (rem-based):

| Token                | ‚Üí scale           | ‚Üí rem      |
| -------------------- | ----------------- | ---------- |
| `--mds-spacing-none` | `--mds-scale-0`   | `0rem`     |
| `--mds-spacing-xxs`  | `--mds-scale-100` | `0.25rem`  |
| `--mds-spacing-xs`   | `--mds-scale-300` | `0.5rem`   |
| `--mds-spacing-sm`   | `--mds-scale-400` | `0.75rem`  |
| `--mds-spacing-md`   | `--mds-scale-600` | `1rem`     |
| `--mds-spacing-lg`   | `--mds-scale-700` | `1.25rem`  |
| `--mds-spacing-xl`   | `--mds-scale-1000`| `2rem`     |
| `--mds-spacing-xxl`  | `--mds-scale-1200`| `3rem`     |

Dimension tokens use rem (not px). Style Dictionary transform `dimension-px-to-rem` divides raw px by 16, excluding font-weight.

### Guardrails (HARD RULES ‚Äî apply to agents too)

1. **Never edit token files.** `tokens/dtcg/**/*.json`, `tokens/css/**`, `tokens/scss/**` are governed by an automated **ZeroHeight PR flow**. Do **not** run `npm run build-tokens` in a plugin consumer.
2. **Never invent an ad-hoc CSS variable or hardcoded value.** If a matching token does not exist, request one at **https://design.moodle.com/**.
3. **Do not add `var(--mds-foo, fallback)` fallbacks.** Tokens are always defined via the `@moodlehq/design-system/css` import chain; a fallback only masks a missing token that should be fixed upstream.
4. Renaming/removing a token is a **breaking change** ‚Äî consumer CSS fails silently (the `var()` resolves to nothing, no error).

## 4. Components ‚Äî rules for consumers and contributors

Every component has this file structure:

```
components/<name>/
  ComponentName.tsx          # implementation
  component-name.css         # colocated styles using --mds-* tokens
  ComponentName.test.tsx     # Vitest unit tests
  ComponentName.stories.tsx  # Storybook stories (also interaction/a11y tests)
  index.tsx                  # barrel: export * from './ComponentName';
```

Components are re-exported from `components/index.tsx`:

```ts
export { Button } from './button';
export type { ButtonProps } from './button';
```

### 4.1 Shipped components (as of v2.1.1)

**Only `Button` is shipped in v2.1.x.** More follow per release. Do not assume other MDS components exist until verified.

#### `<Button>`

```tsx
import { Button } from '@moodlehq/design-system';

<Button
  label={getString('save', 'core')}   // REQUIRED (or aria-label for icon-only)
  variant="primary"                   // 'primary' | 'secondary' | 'danger'
                                      // 'outline-primary' | 'outline-secondary' | 'outline-danger'
  size="sm"                           // 'sm' | 'lg'   (no 'md' ‚Äî omit for default)
  startIcon={<i className="fa fa-save" aria-hidden="true" />}
  endIcon={<svg>‚Ä¶</svg>}
  onClick={handler}
  data-testid="save-btn"              // arbitrary native / aria / data attrs pass through via {...props}
/>
```

**Hard rules (enforce on every consumer call site):**

- **Pass text via `label`, NOT via `children`.** Children are silently ignored. Icon-only buttons must still provide `aria-label`.
- **Never hardcode English strings as `label`.** The caller owns i18n: `label={getString('save','core')}` (PHP-rendered) or `label={t('core:save')}` (React app context). Raw literals bypass translation.
- **Icon props accept only `<i>` or `<svg>` elements.** Anything else is rejected with a dev-mode `console.error` and not rendered.
- **Invalid variant silently falls back to `'primary'`** with a dev-mode warning. In production builds the fallback is silent ‚Äî do not rely on it.
- **Disabled state goes through the native `disabled` attribute** (spread through `...props`).

### 4.2 Class-name layering ‚Äî why two classes

The rendered DOM always carries **both** `mds-btn` and Bootstrap's `btn` (plus `btn-<variant>`). The `mds-*` hook is the override surface; the `.btn` class supplies Bootstrap's structural CSS. Component CSS uses **three specificity levels** ‚Äî pick the right one:

```css
/* 1. Base ‚Äî double-class (0,2,0) overrides Bootstrap's single-class .btn */
.mds-btn.btn {
  background-color: var(--mds-bg-interactive-primary-default);
  border-radius: var(--mds-border-radius-md);
  padding: var(--mds-spacing-xs) var(--mds-spacing-sm);
  font-size: var(--mds-font-size-paragraph-default);
}

/* 2. Variant ‚Äî double-class, same specificity as base, wins by cascade order */
.mds-btn.btn-danger {
  background-color: var(--mds-bg-interactive-danger-default);
}

/* 3. State on variant ‚Äî triple-class (0,3,0) beats both base and variant rules */
.mds-btn.btn.btn-danger:disabled {
  background-color: var(--mds-bg-interactive-danger-disabled);
}
```

**Never use `!important`.** Resolve conflicts through specificity or cascade order only.

### 4.3 API contract rules (contributors)

| Rule | Why |
| ---- | --- |
| Props extend native HTML attr interfaces (`ButtonHTMLAttributes<HTMLButtonElement>`), never third-party types | Keeps types stable across library changes |
| Use `interface` for props, not `type`. `import type` for type-only imports | Convention + bundler-friendly |
| Constrained props use a runtime `allowedValues` array + dev-warning + safe fallback | TS alone doesn't protect JS consumers |
| Named re-exports (`export { Button }`), not `export { default as Button }` in barrels | `preserveModules` requires named exports |
| Class composition: `['mds-<name>', 'btn', `btn-${resolvedVariant}`, ...size, ...className]` ‚Äî `mds-*` first, consumer `className` last | Specificity layering + consumer override |
| Spread `...props` on the root element **last** | Lets consumers pass `aria-*`, `data-*`, handlers without explicit forwarding |

### 4.4 Breaking vs non-breaking changes

**Breaking (require major version bump):**

- Removing or renaming a prop
- Narrowing a prop's type (`string` ‚Üí `'a' | 'b'`)
- Changing a prop's default behavior
- Removing or renaming an export
- Removing/renaming a value in `allowedVariants` (silent fallback breaks consumer expectation)
- Changing the root element type (`<button>` ‚Üí `<div>` breaks CSS selectors, ARIA, event forwarding)

**Safe:** adding an optional prop with default, widening a type, adding a new export, adding a value to `allowedVariants`, adding a new token.

### 4.5 Compound components

Components with distinct regions (e.g. `Card` with `CardHeader`, `CardBody`) export each sub-component by name ‚Äî **never** use dot-notation (`Card.Header`). Named exports are tree-shakeable and match the barrel convention:

```ts
// components/card/index.tsx
export { Card } from './Card';
export { CardHeader } from './CardHeader';
export { CardBody } from './CardBody';
```

### 4.6 i18n & RTL

- **Strings:** every user-visible text is a prop. Caller translates. No hardcoded English.
- **CSS logical properties:** use `margin-inline-start`/`margin-inline-end`, `padding-inline-*`, `inset-inline-*`, `border-inline-*`, `text-align: start/end`. Do NOT use `left`/`right` or `margin-left`/`margin-right` for directional styling.
- **`dir` attribute:** inherited from the document/ancestor; consumers can pass `dir` directly since `...props` is spread.
- **Locale-aware formatting:** MDS does not format dates/numbers/currency. Consumers pre-format.

### 4.7 Agent guardrails (components)

- **Do not remove or rename exports from `components/index.tsx`.** Story/test files are excluded from `tsconfig.json`, so removing a public export does NOT break `npm run build` ‚Äî it breaks downstream consumers silently.
- **Do not add icon or image packages.** Use SVG/image assets from Figma MCP (`get_design_context`, `get_screenshot`).
- **Do not edit `.storybook/preview.ts` a11y setup or theme decorator.** Preserve the existing configuration.

## 5. Integration recipes for eLeDia plugins

### 5.1 Moodle 5.2+ plugin (React + MDS)

Inside `js/react/src/**/*.tsx`:

```tsx
import { Button } from '@moodlehq/design-system';
import { getString } from 'core/str';

export const SaveBar = ({ onSave }: { onSave: () => void }) => (
  <div className="d-flex gap-2 flex-wrap">
    <Button label={getString('save','core')} variant="primary" onClick={onSave} />
    <Button label={getString('cancel','core')} variant="outline-secondary" />
  </div>
);
```

CSS: the `@moodlehq/design-system` JS-Paket is served to the browser via Moodle's import-map (`core\output\requirements\import_map`). The tokens & component CSS are provided by Moodle-core's layout. No app-level `import '@moodlehq/design-system/css'` is needed inside plugin code.

### 5.2 SCSS-only consumer (no React; custom plugin styles reading tokens)

Modern (`@use`):

```scss
@use '@moodlehq/design-system/tokens/scss' as *;

.my-plugin-card {
  background: var(--mds-bg-surface-default);
  border: var(--mds-stroke-weight-sm) solid var(--mds-border-subtle);
  border-radius: var(--mds-border-radius-md);
  padding: var(--mds-spacing-md);
  box-shadow: var(--mds-shadow-lg);
  color: var(--mds-text-default);
}
```

Legacy (`@import`) ‚Äî **only** when building inside Moodle Core LMS or another `scssphp`-bound environment:

```scss
@import '@moodlehq/design-system/tokens/scss/legacy';

.my-plugin-card {
  background: var(--mds-bg-surface-default);
  /* ‚Ä¶ */
}
```

> SCSS tokens are emitted as **flat values**, not variable references (Style Dictionary's `scssVariables` formatter limitation ‚Äî a custom formatter would need to handle hoisting/sorting). This is intentional; do not treat it as a bug.

### 5.3 Multi-version plugin (4.5 ‚Äì 5.2)

For a plugin supporting `$plugin->supported = [405, 502]`:

- Prefer Bootstrap 5 utility classes that Moodle 4.5 Boost and 5.2 both understand.
- Reference `--mds-*` variables only inside a `@supports (--mds-bg-interactive-primary-default: 0)` guard, or ship a Boost-4.5 fallback SCSS that defines the minimal `--mds-*` values used.
- Do **not** import `@moodlehq/design-system` React components from a 4.5 build ‚Äî React is only a first-class citizen in 5.2+.

### 5.5 Button variant migration

| Intent                        | Old assumption (WRONG)                              | Real MDS Button API |
| ----------------------------- | --------------------------------------------------- | ------------------- |
| Primary action                | `<Button variant="primary">‚Ä¶</Button>`              | `<Button variant="primary" label={t(...)} />` |
| Alternative action            | `<Button variant="secondary">‚Ä¶</Button>`            | `<Button variant="secondary" label={t(...)} />` or `variant="outline-secondary"` |
| Destructive action            | `<Button variant="destructive">‚Ä¶</Button>`          | `<Button variant="danger" label={t(...)} />` |
| Small outline-danger          | `<Button size="md">‚Ä¶</Button>`                      | `<Button variant="outline-danger" size="sm" label={t(...)} />` |
| Default size (medium)         | `size="md"`                                         | **Omit `size`** ‚Äî there is no `'md'`. Only `'sm'` and `'lg'` |
| Text via children             | `<Button>Save</Button>`                             | `<Button label={t('core:save')} />` |

## 6. Testing & development (contributors)

### 6.1 Prerequisites

```bash
node --version    # must be ‚â• v22.13.0
npm install
npx playwright install   # required for Storybook browser tests
```

### 6.2 Commands

| Command                            | Purpose |
| ---------------------------------- | ------- |
| `npm run storybook`                | Dev server at http://localhost:6006 |
| `npm run test-unit`                | Vitest + jsdom for `components/**/*.{test,spec}.*` |
| `npm run test-unit-coverage`       | + Istanbul coverage (50% min, 80% target statement coverage) |
| `npm run test-storybook`           | Storybook stories tagged `test` via Playwright/Chromium |
| `npm run build`                    | `tsc && vite build` ‚Üí `dist/` |
| `npm run build-storybook`          | Static Storybook output |
| `npm run build-tokens`             | **Humans only.** Regenerates `tokens/css/*` and `tokens/scss/*` from DTCG sources |
| `npm run lint` / `npm run format`  | ESLint + Prettier (configs under `.github/linters/`) |

Run a single unit test file:

```bash
VITEST_STORYBOOK=false npx vitest run components/button/Button.test.tsx
```

The `VITEST_STORYBOOK` env var switches modes: `false` ‚Üí jsdom unit tests; `true` ‚Üí Storybook stories tagged `test` via Playwright/Chromium headless.

### 6.3 Stories ‚Äî CSF rules

- Default tags for a new story: `tags: ['autodocs', 'test', 'stable']`.
- Add an `experimental` tag to exclude from Storybook Vitest runs.
- For any directional layout, add an RTL story (tags `['test','stable']`, omit `autodocs`):

  ```tsx
  export const RightToLeft: Story = {
    decorators: [(Story) => (<div dir="rtl"><Story /></div>)],
  };
  ```

**Play functions ‚Äî two things that silently fail if wrong:**

1. **Use `canvas`, not `screen`.** `canvas` is scoped to the story's DOM; `screen` is global.
2. **Import `expect` from `'storybook/test'`**, never from `'vitest'`. They look identical; only the Storybook one works inside a `play` function.

Flush microtasks after an interaction if asserts flake:

```ts
await new Promise(r => setTimeout(r, 0));
```

### 6.4 Unit-test baseline for a new component

Cover at minimum:

- `mds-*` class applied to root element
- Each variant / size renders expected classes
- Invalid variant prop ‚Üí falls back to default
- Extra props forwarded (`data-testid`, `aria-label`, etc.)
- Disabled state behaves correctly
- Label/content renders as expected

Use property-based fuzzing (`tests/utils/fuzzComponent.ts` + `fast-check`) only for components with 3+ independent props each with multiple values. For small-variant components, standard `it` tests are preferred.

**Never `skip`/`skip.test` or delete a `play` function to make a test pass.** If skipping is genuinely required (upstream bug), add an inline comment with the reason + ticket.

### 6.5 Accessibility

Every story tagged `test` is automatically scanned by **Axe** against **WCAG 2.2 AA** and best-practice rules (configured in `.storybook/preview.ts`). Violations fail the build. Do not disable a11y checks without a documented reason.

## 7. Commit & contribution rules (if opening a PR against moodlehq/design-system)

- **Conventional Commits** required. Allowed types: `build`, `chore`, `ci`, `docs`, `feat`, `fix`, `perf`, `refactor`, `revert`, `style`. Subject uses **sentence case**.
- Husky + lint-staged runs Prettier + ESLint on commit. Do not bypass hooks.
- Releases are automated via **Release Please** ‚Äî never manually edit `CHANGELOG.md` or `package.json` version fields.

## 8. MCP servers useful in MDS work

If contributing to MDS or pulling design intent into a plugin, these MCP servers are documented in the repo's `copilot-instructions.md`:

| Server              | Purpose |
| ------------------- | ------- |

| **Storybook MCP**   | `http://localhost:6006/mcp` when `npm run storybook` is running ‚Äî query rendered stories |
| **chrome-devtools MCP** | Inspecting computed styles of rendered output |
| **GitHub MCP**      | Reading issues/PRs/CI status |

When working from a Figma design:

1. Fetch `get_design_context` (structure) and `get_screenshot` (visual reference) first.
2. If the payload is large, use `get_metadata` to narrow, then re-fetch the target node.
3. Treat Figma output as a **reference** ‚Äî translate to existing component patterns + `--mds-*` tokens, not final code.
4. Use SVG/image assets from Figma MCP directly; don't add new packages.

## 10. Quick sanity-check checklist for any MDS-touching PR

- [ ] All colors, spacings, radii, shadows, typography come from `var(--mds-*)` ‚Äî no hardcoded values
- [ ] No `!important`
- [ ] No `var(--mds-foo, fallback)` ‚Äî tokens are always defined
- [ ] `mds-*` class is first in the `className` list; any consumer `className` is last
- [ ] Component text goes through `label` / a named string prop ‚Äî not children, not raw English literals
- [ ] Icons are `<i>` or `<svg>` elements only
- [ ] CSS uses logical properties (`inline-start`/`inline-end`) for any directional styling
- [ ] No edits to `tokens/dtcg/**`, `tokens/css/**`, `tokens/scss/**`, `CHANGELOG.md`, version fields, or workflow files
- [ ] Conventional Commit subject in sentence case
